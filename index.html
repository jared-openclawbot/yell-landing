Yell is not a template engine. It is a UI runtime for teams who want component trees that are inspectable, portable, serializable, and governable.

Built for product surfaces where systems matter more than syntax preference: internal tools, AI-generated views, content-heavy SaaS, and design-system-led platforms.

Why teams are looking beyond JSX

Structured UI that plays well with infra, not just developers.

Serializable syntax

YAML gives you a transportable interface definition you can persist, cache, diff, validate, and version without inventing a parallel DSL.

Design-system boundaries

Restrict surfaces to approved components, tokens, slots, and prop shapes instead of exposing arbitrary render logic everywhere.

SSR + hydration

Render from schema on the server, hydrate only what is interactive, and keep the runtime aligned with a stable component contract.

AI-native interfaces

LLMs are dramatically better at producing structured, validated interface trees than ad-hoc JSX blobs with business logic sprinkled everywhere.

Governed composition

Let infra teams define what can be rendered, who can generate it, and how it gets validated before it ever reaches a browser.

Production observability

Trace UI definitions as data: inspect them in logs, snapshot them in storage, and replay them across environments with fewer unknowns.

The pitch

What if your frontend syntax was also your runtime contract?

Yell treats interface definitions as structured documents, not arbitrary code islands. That makes it unusually good at powering generated UIs, controlled composition, server rendering, and multi-team design-system workflows.

  • Strict schema validation before render
  • Portable config that survives services, queues, and caches
  • Clean separation between view declaration and imperative behavior
Why not JSX?

JSX is flexible, expressive, and beloved — but it is also harder to serialize, harder to govern, harder to diff semantically, and awkward when your UI needs to be generated or transported as data.

Yell is for the class of products where structure beats cleverness and where interfaces increasingly need to move between humans, services, and models.

Platform capabilities

FAQ

Reasonable questions from sensible people.

Is this replacing React?

No. Yell is a declarative rendering layer and runtime contract. It can target familiar frontend ecosystems while changing how UI is expressed and governed.

Can YAML really handle modern application UI?

Yes, if the runtime is built around typed components, event contracts, server rendering, and strict design-system boundaries rather than raw template interpolation.

Is this only for AI-generated interfaces?

No — but AI is a compelling reason to want structured, serializable UI. The same benefits help internal tools, CMS-driven surfaces, and systemized product shells.

What about hydration and interactivity?

Yell keeps interactivity explicit, so the runtime can hydrate only what is needed instead of reconstituting a giant opaque tree.
